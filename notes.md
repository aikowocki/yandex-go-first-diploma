# Заметки по архитектуре

## Advisory Lock vs FOR UPDATE

### pg_advisory_xact_lock(key)
- Абстрактный мьютекс в PostgreSQL по числовому ключу
- Не привязан к таблицам или строкам
- `xact` — живёт до конца транзакции (commit/rollback), отпускается автоматически
- Одноключевая версия принимает `bigint`, двухключевая — два `int` (32-bit)
- Двухключевая: `pg_advisory_xact_lock(1, userID)` — первый ключ как namespace, но `int4` может переполниться

### SELECT ... FOR UPDATE
- Блокирует конкретную строку в таблице
- Для ledger-подхода (баланс = SUM) нет строки для блокировки → создают таблицу `user_locks`
- Стандартный механизм, видно в `pg_locks`
- Нужно не забыть создать строку при регистрации пользователя

### Когда что
- Advisory lock — проще, не нужна лишняя таблица, подходит для дипломного проекта
- FOR UPDATE на lock-таблице — продакшен-подход для ledger-only систем
- UPDATE balances SET current = current - $1 WHERE current >= $1 — продакшен-подход с кэшированным балансом

## Баланс: Ledger-only vs Cached Balance

### Ledger-only (наш подход)
- Баланс = `SUM(accrual) - SUM(withdrawal)` по таблице transactions
- Единственный источник правды, не может разойтись
- Используется в финтехе для аудита и сверки
- Минус: SUM() на каждый запрос, медленнее при большом количестве транзакций

### Cached Balance (продакшен)
- Отдельная колонка/таблица `balances` с текущим значением
- `UPDATE balances SET current = current - $1 WHERE current >= $1` — атомарно, без lock'ов
- Transactions остаются как лог/аудит
- Минус: баланс может разойтись с суммой транзакций (баг, ручная правка)

## Кэширование баланса в Dragonfly/Redis

### Паттерн "Декоратор"
- Новый адаптер `CachedBalanceRepo` оборачивает `BalanceRepository`
- Usecase и handler не меняются — работают через тот же интерфейс
- В main.go подставляем обёртку вместо оригинального репо

### Структура файлов
```
internal/adapter/cache/
  client.go    — подключение к Dragonfly
  balance.go   — декоратор CachedBalanceRepo
```

### Инвалидация
- После Withdraw и AddAccrual — `DEL balance:{userID}`
- Следующий GetBalance промахнётся → пойдёт в базу → положит в кэш
- TTL обязателен (например 5 мин) — страховка если DEL упадёт
- Ошибку DEL логируем, но не роняем операцию — списание важнее кэша

### Кэш только для чтения
- GET /balance → из кэша
- Withdraw → всегда через PostgreSQL под lock'ом, потом инвалидация кэша
- Нельзя проверять баланс по кэшу перед списанием — race condition
